# Core

Generic ESLint-compatible rules that apply to both JavaScript and TypeScript source. Every rule listed here corresponds 1-to-1 with an ESLint core rule of the same kebab-case id, so projects migrating from ESLint can paste their rule severities into `lint.config.ts` without renaming anything. TypeScript-only rules and `@typescript-eslint` extensions live under `typescript/*` in [TypeScript](./typescript).

Source: [ESLint core rules](https://eslint.org/docs/latest/rules/) (MIT).

- `default-param-last` ‚Äî Reject `(req, opt = 1, req2)` and similar parameter lists where a required parameter follows an optional or default-valued one.
- `dot-notation` ‚Äî Prefer dot access (`obj.value`) over bracket access (`obj["value"]`) when the string key is a valid JavaScript identifier.
- `eqeqeq` ‚Äî Require strict equality operators `===` / `!==` over `==` / `!=`.
- `for-direction` ‚Äî Reject `for` statements whose update clause moves the counter away from the termination condition, such as `for (let i = 0; i < 10; i--)`.
- `getter-return` ‚Äî Require a `get` accessor's body to return a value on every reachable exit.
- `no-alert` ‚Äî Reject calls to `alert`, `confirm`, and `prompt`.
- `no-array-constructor` ‚Äî Reject `Array(...)` and `new Array(...)` constructor calls in favor of array literals.
- `no-async-promise-executor` ‚Äî Reject `new Promise(async (resolve, reject) => { ... })`.
- `no-await-in-loop` ‚Äî Reject `await` expressions evaluated inside a loop body.
- `no-bitwise` ‚Äî Reject bitwise operators (`&`, `|`, `^`, `~`, `<<`, `>>`, `>>>`).
- `no-caller` ‚Äî Reject `arguments.caller` and `arguments.callee`, both deprecated properties forbidden in strict mode.
- `no-case-declarations` ‚Äî Reject lexical declarations (`let`, `const`, `class`, `function`) inside `case` or `default` clauses without their own block, since the declaration shares the whole `switch` scope and leaks into sibling clauses.
- `no-class-assign` ‚Äî Reject reassigning a class binding (`class C {}; C = ...`).
- `no-compare-neg-zero` ‚Äî Reject comparisons against `-0` (`x === -0`, `x < -0`, etc.).
- `no-cond-assign` ‚Äî Reject assignment expressions inside conditions, such as `if (x = y)` ‚Äî almost always a typo for `==` / `===`.
- `no-console` ‚Äî Reject calls to `console.*`.
- `no-constant-condition` ‚Äî Reject conditions whose value can be determined statically, such as `while (true)` or `if (false)`, in `if`, `while`, `do/while`, `for`, and ternary expressions.
- `no-constructor-return` ‚Äî Reject `return X;` (with a value) inside a class constructor.
- `no-continue` ‚Äî Reject `continue` statements.
- `no-control-regex` ‚Äî Reject ASCII control characters (`\x00`‚Äì`\x1F`) inside regular expression literals and `RegExp` strings.
- `no-debugger` ‚Äî Reject `debugger` statements.
- `no-delete-var` ‚Äî Reject `delete` applied to plain variable bindings (`delete x`).
- `no-dupe-args` ‚Äî Reject `function f(a, a)` and similar parameter lists that declare the same name twice.
- `no-dupe-class-members` ‚Äî Reject two declarations of the same member on a single class.
- `no-dupe-else-if` ‚Äî Reject `if (a) {} else if (a) {}` ‚Äî the second branch is unreachable because the first condition already handled it.
- `no-dupe-keys` ‚Äî Reject `{ a: 1, a: 2 }` ‚Äî duplicate property keys in an object literal silently overwrite earlier values.
- `no-duplicate-case` ‚Äî Reject the same `case` label appearing twice in a `switch` ‚Äî later duplicates are unreachable.
- `no-duplicate-imports` ‚Äî Reject two import declarations that resolve to the same module specifier.
- `no-empty` ‚Äî Reject empty blocks (`if (x) {}`, `while (x) {}`, `try {} catch (e) {}` etc.) that almost always indicate forgotten code.
- `no-empty-character-class` ‚Äî Reject empty regex character classes (`[]`).
- `no-empty-function` ‚Äî Reject empty function and method bodies (`function f() {}`, `() => {}`).
- `no-empty-pattern` ‚Äî Reject empty destructuring patterns (`const {} = obj`, `function f([]) {}`), which bind nothing and are usually mid-edit typos.
- `no-empty-static-block` ‚Äî Reject empty `static {}` class initialization blocks.
- `no-eq-null` ‚Äî Reject loose null comparisons (`x == null`).
- `no-eval` ‚Äî Reject `eval(...)` and indirect `eval` calls ‚Äî almost always a security or correctness bug.
- `no-ex-assign` ‚Äî Reject reassigning the parameter of a `catch` clause (`catch (e) { e = ... }`), which loses the original error reference.
- `no-extra-bind` ‚Äî Reject unnecessary `Function.prototype.bind()` calls ‚Äî for example, binding without arguments or binding an arrow function (which ignores `this`).
- `no-extra-boolean-cast` ‚Äî Reject redundant boolean casts such as `!!Boolean(x)`, `if (Boolean(x))`, or `Boolean(!!x)`.
- `no-fallthrough` ‚Äî Reject `switch` case fall-through unless preceded by an explicit `// falls through` comment.
- `no-func-assign` ‚Äî Reject reassignment of function declarations (`function f() {}; f = 0;`).
- `no-implicit-coercion` ‚Äî Reject common implicit-coercion idioms (`!!x`, `+x`, `"" + x`) in favor of the explicit `Boolean(x)` / `Number(x)` / `String(x)` conversions.
- `no-inner-declarations` ‚Äî Reject `function` and `var` declarations nested in non-function blocks (loops, `if`, etc.) ‚Äî they hoist in surprising ways.
- `no-irregular-whitespace` ‚Äî Reject irregular whitespace characters (zero-width space, non-breaking space, etc.) in source ‚Äî typically copy-paste artifacts from rich-text editors.
- `no-iterator` ‚Äî Reject the legacy `__iterator__` property ‚Äî a SpiderMonkey-only extension predating ES2015 iterators.
- `no-labels` ‚Äî Reject labeled statements (`outer: for (...) { break outer; }`).
- `no-lone-blocks` ‚Äî Reject standalone `{ ... }` blocks that introduce no lexical scope distinct from the surrounding block.
- `no-lonely-if` ‚Äî Reject `if (cond) { if (...) { ... } }` where the inner `if` is the only statement in an `else` ‚Äî prefer `else if`.
- `no-loss-of-precision` ‚Äî Reject numeric literals whose source text cannot round-trip through `Number` without losing digits, including overflow.
- `no-misleading-character-class` ‚Äî Reject regex character classes that contain combined Unicode sequences (e.g. surrogate pairs) which most readers will not realize represent multiple code units.
- `no-mixed-operators` ‚Äî Reject mixing operators of different precedence families (`a && b || c`, `a | b && c`) without explicit parentheses around the inner sub-expression.
- `no-multi-assign` ‚Äî Reject chained assignment such as `a = b = 0`, which obscures intent and surprises readers who expect comparison.
- `no-multi-str` ‚Äî Reject backslash-newline multiline string literals; use template literals instead.
- `no-negated-condition` ‚Äî Reject `if (!cond) { ... } else { ... }` ‚Äî flip the branches so the positive condition reads first.
- `no-nested-ternary` ‚Äî Reject ternary expressions nested in other ternaries (`a ? b : c ? d : e`), which are hard to read at a glance.
- `no-new` ‚Äî Reject `new` expressions whose return value is not assigned or used ‚Äî the object is created only for its constructor side effects.
- `no-new-func` ‚Äî Reject `new Function(...)` and `Function(...)` calls, which effectively evaluate a string and have the same risks as `eval`.
- `no-new-symbol` ‚Äî Reject `new Symbol(...)`.
- `no-new-wrappers` ‚Äî Reject primitive wrapper constructors `new String(...)`, `new Number(...)`, `new Boolean(...)`.
- `no-obj-calls` ‚Äî Reject calling global non-callable objects as functions, such as `Math()` or `JSON()`.
- `no-object-constructor` ‚Äî Reject `new Object()` and `Object()` constructor calls; use an object literal `{}` instead.
- `no-octal` ‚Äî Reject legacy octal literals (`0123`).
- `no-octal-escape` ‚Äî Reject octal escape sequences in string literals (`"\251"`, `"\07"`).
- `no-plusplus` ‚Äî Reject `++` and `--` operators.
- `no-promise-executor-return` ‚Äî Reject `return` inside the Promise executor function ‚Äî the value is ignored.
- `no-proto` ‚Äî Reject access to `obj.__proto__`; use `Object.getPrototypeOf` / `Object.setPrototypeOf`.
- `no-prototype-builtins` ‚Äî Reject `obj.hasOwnProperty(key)` and other direct `Object.prototype` builtins on user objects, since the property may be shadowed.
- `no-regex-spaces` ‚Äî Reject more than one consecutive literal space in a regex; use `{N}` quantifiers for clarity.
- `no-return-assign` ‚Äî Reject assignment expressions used as the operand of `return` (`return x = 1`) ‚Äî almost always a typo for `===`.
- `no-script-url` ‚Äî Reject `javascript:` URLs in string literals ‚Äî they execute their body as code on browser navigation, and security scanners treat them as an `eval` equivalent.
- `no-self-assign` ‚Äî Reject `x = x` and destructuring forms that copy a value to itself ‚Äî almost always a typo.
- `no-self-compare` ‚Äî Reject comparing a value to itself (`x === x`).
- `no-sequences` ‚Äî Reject comma expressions (`a, b`) outside the heads of `for` statements.
- `no-setter-return` ‚Äî Reject explicit `return` from a setter ‚Äî setters' return values are ignored.
- `no-shadow-restricted-names` ‚Äî Reject redeclaring restricted globals (`NaN`, `Infinity`, `undefined`, etc.).
- `no-sparse-arrays` ‚Äî Reject array literals with elision (`[, 1, , 3]`), which read surprisingly and rarely express intent.
- `no-template-curly-in-string` ‚Äî Reject `${expr}` inside ordinary single- or double-quoted strings ‚Äî almost always a missing template-literal backtick.
- `no-this-before-super` ‚Äî Reject `this` (or `super.x`) references that precede the first `super()` call in a derived constructor.
- `no-throw-literal` ‚Äî Reject throwing non-Error operands (`throw "boom"`, `throw 1`).
- `no-undef-init` ‚Äî Reject initializing a variable to the literal `undefined` (`let x = undefined`) ‚Äî declaring without an initializer has the same effect.
- `no-undefined` ‚Äî Reject use of the global `undefined` identifier; use the `void 0` expression or omit the value.
- `no-unneeded-ternary` ‚Äî Reject `cond ? true : false` and similar ternaries that can be simplified to a boolean coercion or the condition itself.
- `no-unsafe-finally` ‚Äî Reject `return` and `throw` inside a `finally` block, which override any earlier `return`/`throw` from the corresponding `try`/`catch`.
- `no-unsafe-negation` ‚Äî Reject `!key in obj` and `!a instanceof B` where the `!` binds tighter than the relational operator and silently coerces the left operand to a boolean.
- `no-unsafe-optional-chaining` ‚Äî Reject member access or call expressions that chain off an optional chain without continuing the chain.
- `no-unused-expressions` ‚Äî Reject expression statements with no observable effect, like a bare `x;` or `'use strict' && f();`.
- `no-unused-labels` ‚Äî Reject labels that no `break` or `continue` statement references.
- `no-useless-call` ‚Äî Reject unnecessary `.call()` / `.apply()` calls (such as `f.call(undefined, x)`).
- `no-useless-catch` ‚Äî Reject `catch (e) { throw e }` patterns that only rethrow the caught error without adding context or handling.
- `no-useless-computed-key` ‚Äî Reject computed property keys whose expression is a literal identifier (`{ ["foo"]: 1 }`).
- `no-useless-concat` ‚Äî Reject `"a" + "b"` and similar concatenations where every operand is a literal string.
- `no-useless-constructor` ‚Äî Reject empty constructor bodies (`class X { constructor() {} }`) that add nothing over the implicit constructor.
- `no-useless-escape` ‚Äî Reject unnecessary escape sequences in strings and regex literals, such as `"\."` or `/\,/`.
- `no-useless-rename` ‚Äî Reject `{ x: x }` destructuring renames that bind back to the same name.
- `no-var` ‚Äî Reject `var` declarations.
- `no-with` ‚Äî Reject `with (...)` statements.
- `object-shorthand` ‚Äî Reject `{ foo: foo }` and similar object-literal shorthand candidates in favor of `{ foo }`.
- `operator-assignment` ‚Äî Prefer compound assignment (`x += y`) over the long form (`x = x + y`) where the two are equivalent.
- `prefer-const` ‚Äî Require `const` for variables that are never reassigned after declaration.
- `prefer-exponentiation-operator` ‚Äî Prefer the `**` operator over `Math.pow(base, exp)`.
- `prefer-for-of` ‚Äî Prefer `for..of` over a traditional `for (let i = 0; i < arr.length; i++)` loop when the index is never used inside the body.
- `prefer-numeric-literals` ‚Äî Prefer ES2015+ numeric literal forms (`0b‚Ä¶`, `0o‚Ä¶`, `0x‚Ä¶`) over `parseInt(string, 2 | 8 | 16)`.
- `prefer-object-has-own` ‚Äî Prefer `Object.hasOwn(obj, key)` over `Object.prototype.hasOwnProperty.call(obj, key)`.
- `prefer-object-spread` ‚Äî Prefer object-spread `{ ...a, ...b }` over `Object.assign({}, a, b)`.
- `prefer-spread` ‚Äî Prefer spread arguments `f(...args)` over `f.apply(null, args)`.
- `prefer-template` ‚Äî Prefer template literals over string concatenation when any operand is non-literal.
- `radix` ‚Äî Require an explicit radix argument for `parseInt(str, radix)`.
- `require-yield` ‚Äî Require generator functions to contain at least one `yield`.
- `use-isnan` ‚Äî Require `Number.isNaN` / `isNaN` for `NaN` checks; restrict `typeof` comparisons to the documented strings.
- `valid-typeof` ‚Äî Restrict the right-hand operand of `typeof` to the documented strings (`"number"`, `"object"`, ...) so `typeof x === "undefiend"` typos are caught.
- `vars-on-top` ‚Äî Require `var` declarations to be hoisted to the top of their scope by hand, mirroring how the engine treats them.
- `yoda` ‚Äî Reject Yoda-style comparisons (`if (42 === x)`); use `if (x === 42)` so the variable comes first.
