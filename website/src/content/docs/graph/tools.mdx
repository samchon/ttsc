# Tools

`@ttsc/graph` exposes four MCP tools. In a fresh project session, run `query_exports` once before the first `query_nodes`, then move to the graph queries once you know the public surface. `query_nodes` is for focused relationship and flow questions after that orientation, or when you already have exact symbols or handles. An agent calls them on its own; you never invoke them by hand. They all answer from the resident `Program`, so the result is the compiler's own, at the same code and location `ttsc` reports.

Every tool returns the same MCP content envelope: one or more text blocks. The text is the rendered graph view; the structure is always this shape.

```ts
interface ToolResult {
  content: {
    type: "text";
    /** The rendered graph view for one block (see each tool below). */
    text: string;
  }[];
}
```

## query_exports

Use `query_exports` first on a new project. It gives you the compiler-known exported surface before you ask focused graph questions.

```ts
interface QueryExportsArguments {
  /**
   * Optional filter over exported symbols.
   */
  query?: string;

  /** Maximum export entries to return. Defaults to 100, minimum 0, maximum 10000; 0 returns only the folder/kind summary. */
  limit?: number;

  /** Zero-based offset for the next page. Defaults to 0, minimum 0. */
  offset?: number;
}
```

It returns one text block. By default, the block shows a compact first page of exported symbols plus the full folder/kind summary, then an `Exports:` pagination line, an `Export folders:` summary, and an `Exported symbols:` section. Raise `limit` for a larger page, or use `offset` for the next page. Set `limit` to `0` when you only need the folder/kind summary.

Each exported symbol line includes `kind`, `name`, `exports:<comma list>`, `file:line`, `handle`, `calls:<n> uses:<n> types:<n> deps:<n>`, and `sources:<semicolon list>`.

It covers top-level exported symbols and the public members of exported classes and interfaces. Git-ignored generated files are omitted, and source bodies are not returned.

```ts
type QueryExportsResult = ToolResult; // content has exactly one text block
```

## query_nodes

Use `query_nodes` after the first `query_exports` pass, or when you already know the exact symbols or handles you want to inspect. Its one argument is the query:

```ts
interface QueryNodesArguments {
  /**
   * A symbol name, a file-path fragment, or the salient nouns of a
   * "how does X work" question. Name the whole flow in one broad query
   * rather than one symbol at a time, owner / action / nouns together,
   * e.g. "repository find manager query builder".
   */
  query: string;
}
```

It returns a single text block. That block renders each matched node with:

- the verbatim, line-numbered source of the declaration, capped per node by a budget that scales with how many symbols the query names;
- its checker-resolved edges in both directions: `heritage`, `value-call`, `type-ref` (see [Edge kinds](#edge-kinds));
- its blast radius, the transitive dependents a change would reach;
- its own diagnostics, and how many of those dependents are already broken.

Nodes past the source budget collapse to a one-line signature, so one call never floods the context.

```ts
type QueryNodesResult = ToolResult; // content has exactly one block
```

One call is usually enough context to answer, instead of fanning out to read files.

A query can name several symbols at once. The server ranks matches by name match and edge centrality, so a flow question such as "how is a payment charged" surfaces the cluster of declarations involved, not a single literal hit. The response is bounded, an edge cap, a per-node source cap, and a total budget that scales with how many symbols the query names, so a broad match returns a few kilobytes rather than the whole project.

## query_files

Render one or more files in full. Its one argument is the list of locations:

```ts
interface QueryFilesArguments {
  /**
   * One or more source files, each an absolute path or a trailing
   * fragment of one, e.g. ["src/main.ts", "src/foo.ts"].
   */
  locations: string[];
}
```

It returns one text block per location, in input order, each headed with the location. A file is the objects inside it, so each block renders every declaration in the file with the same `query_nodes`-grade view: its edges, its diagnostics, its blast radius, and its verbatim line-numbered body. This is not a signature outline; the point of a file query is the declarations and how they relate, plus their code. Declarations past the response budget collapse to a one-line signature so one call never floods the context.

```ts
type QueryFilesResult = ToolResult; // one block per requested location, in input order
```

## query_diagnostics

Both arguments are optional:

```ts
interface QueryDiagnosticsArguments {
  /**
   * One or more source files, each an absolute path or a trailing
   * fragment of one, e.g. ["src/main.ts"]. Omit it for the whole project.
   */
  files?: string[];

  /** Keep only `error`, only `warning`, or `all` findings. Defaults to `error`. */
  severity?: "error" | "warning" | "all";
}
```

With `files`, it returns one text block per file, in input order, each headed with the location, at the same code and location `ttsc` reports. With `files` omitted, it returns a single block listing every current diagnostic across the project, grouped by file, for a "what is broken now" check after an edit. Each finding carries its location and, for a tsc error, its `TSxxxx` code.

```ts
type QueryDiagnosticsResult = ToolResult; // one block per file, or a single project-wide block
```

The diagnostics, here and in the errors fused into `query_nodes`, are the project's full set:

- TypeScript type and compile errors.
- `@ttsc/lint` rule violations.
- Transform-plugin errors, from plugins like typia and nestia.

The type errors come from the graph's resident `Program`. The lint and plugin findings come from a background `ttsc` check that runs your project's plugins, fused onto the same graph.

When that background check cannot run (no `ttsc`, no Go toolchain, or a project that does not compile), the graph falls back to the type errors alone.

## Edge kinds

| Kind | Meaning |
| --- | --- |
| `heritage` | A class or interface `extends` / `implements` a base. |
| `value-call` | A declaration or method calls a function, method, or constructor (`new T()`) at runtime. |
| `type-ref` | A declaration mentions a named type in a type position. |

Splitting `value-call` from `type-ref` lets an impact query separate "used at runtime" from "depends on this type".

## Scope

This is a v1. The nodes are the declarations and their methods, including the members of a `namespace`. The edges are the kinds above, resolved by the checker: a method-to-method call, a `new T()` constructor call, and a call from inside a top-level `const fn = () => …` are all edges.

A file watcher and on-disk persistence are not included yet. The node identity is position-invariant, so an incremental layer can be added later without re-keying the graph.

## Large projects

The server answers `initialize` and `tools/list` immediately and builds the `Program` in the background, so the agent always sees the tools; only the first query blocks on the type-check. On a large repository that one-time build can take minutes.

To pay that build once and share it across every session, run a resident daemon:

| Flag | Effect |
| --- | --- |
| `--daemon` | Build the `Program` once and serve many connections over a loopback (`127.0.0.1`) port, instead of repeating the type-check per session. |
| `--connect <host:port>` | Proxy this session's stdio to a running daemon. |
| `--port-file <path>` | Daemon: write the chosen `host:port` here (mode `0600`), so a launcher can find it. |
| `--idle <dur>` | Daemon: exit after this idle period (default `5m`; `0` disables). |

The benchmark walks this path end to end on a roughly 7,000-file repository; see the [large-repository path in the Benchmark](/docs/benchmark/graph#vs-code-the-large-repository-path).

> The daemon's loopback port is **unauthenticated**: any process on the machine that connects to it can read this project's source through the graph, so run the daemon only on a single-user host. The default `--stdio` transport has no port, spawns one server per session, and is what the `@ttsc/graph` launcher always uses; the daemon is opt-in.
