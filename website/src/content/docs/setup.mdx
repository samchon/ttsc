# Setup

Three packages, one `lint.config.ts`, one VS Code extension. Five minutes.

## Requirements

- **Node.js** 18 or later.
- A TypeScript project — anything with a `tsconfig.json` works.

## Install

```bash
npm install -D ttsc @ttsc/lint @typescript/native-preview
```

`pnpm` / `yarn` / `bun` work the same way (substitute `add` for `install`).

`@typescript/native-preview` is the TypeScript-Go preview that `ttsc` runs on. The TypeScript team versions it separately, so it's a sibling package — pin it in your project and `ttsc` picks it up from `node_modules`.

## Minimal `tsconfig.json`

If you don't already have one:

```jsonc
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "strict": true,
    "outDir": "dist",
    "rootDir": "src",
  },
  "include": ["src"],
}
```

Same shape as a `tsc` project — `ttsc` reads the same fields.

## Minimal `lint.config.ts`

Drop this next to `tsconfig.json` so `@ttsc/lint` knows which rules to enable:

```ts
// lint.config.ts
import type { ITtscLintConfig } from "@ttsc/lint";

export default {
  rules: {
    "no-var": "error",
    "prefer-const": "error",
    "no-explicit-any": "warning",
  },
  format: {
    printWidth: 100,
    singleQuote: true,
    trailingComma: "all",
  },
} satisfies ITtscLintConfig;
```

`rules` sets severity per lint rule. `format` is a Prettier-style block — the keys mirror `.prettierrc`. See [Lint & Prettier](/docs/lint) for the full catalog.

## First build

```ts
// src/index.ts
function greet(name: string): string {
  return `hello, ${name} — running on ttsc`;
}

console.log(greet("world"));
```

```bash
npx ttsc            # type-check + lint + emit to dist/
npx ttsc --noEmit   # type-check + lint only (CI form)
npx ttsc --watch    # watch mode
npx ttsc fix        # autofix every fixable lint and format violation
npx ttsc format     # rewrite source files with the format rules
node dist/index.js  # run the emit
```

That's it. From here you can:

- **Run a script directly** — `npx ttsx src/index.ts`. Type-checks, then executes. See [TTSC · Execute](/docs/ttsc/execute).
- **Add a transform plugin** — e.g. [typia](https://typia.io) for runtime validators generated from your types.
- **Wire your editor** — install the VS Code extension below for the project
  TypeScript-Go language server plus LSP-capable ttsc plugin diagnostics.

## Editor (VS Code)

The VS Code extension starts the `ttscserver` launcher from your project's own
`ttsc` dependency, so VS Code uses the same TypeScript-Go language server your
project pins.

Not yet on the VS Code Marketplace. For now it ships as an npm package with a one-shot installer:

```bash
npm install -D @ttsc/vscode
npx ttsc-vscode
```

`npx ttsc-vscode` invokes `code --install-extension` against the `.vsix` bundled inside the package. To uninstall:

```bash
npx ttsc-vscode uninstall
```

What it adds:

- TypeScript-Go hover, navigation, completions, and diagnostics through the
  project-selected `tsgo --lsp --stdio` process.
- LSP-capable ttsc plugin diagnostics, code actions, and executeCommand
  handlers. `@ttsc/lint` contributes lint diagnostics, fix-all actions, and
  document format edits. Plugin diagnostics, code actions, and command
  computation use the saved project state today; commands return VS Code
  `WorkspaceEdit`s and VS Code applies them only while touched documents are
  still clean.
- Multi-root workspaces start server contexts from the active file and
  workspace folders. Add packages as VS Code workspace folders when each
  package should keep its own `ttsc`, `@typescript/native-preview`, and plugin
  config active.
- `ttsc: Restart language server` in the Command Palette.
- `ttsc: Fix all lint issues` and `ttsc: Format document` are always
  registered by the extension, but they return edits only when the selected
  file's server has an LSP-capable owner such as `@ttsc/lint`. Other
  plugin-owned command ids are advertised through `vscode-languageclient`; when
  they return a `changes`-map `WorkspaceEdit`, the extension applies it with the
  same clean-document guard.
- Leave `ttsc.serverPath` empty for plugin diagnostics/actions. The default
  project `ttsc` launcher builds the plugin manifest before starting the native
  server; a direct native binary path is useful only when that manifest is
  supplied by the caller.

If diagnostics don't appear after install, open **View → Output**, select
**ttsc** from the dropdown, and read the server log.

Full reference: [`@ttsc/vscode`](https://github.com/samchon/ttsc/tree/master/packages/vscode).
