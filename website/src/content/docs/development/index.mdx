# Plugin Development

For plugin authors — people writing **new** `ttsc` plugins. If you just want to use existing plugins like `@ttsc/lint` or `typia`, you don't need this section — wire the plugin into `compilerOptions.plugins` per its own README and skip ahead.

> **Reading order for first-time authors:**
> (1) [Concepts overview](/docs/development/concepts) — the two-halves model, vocabulary, stages, auto-discovery (5 min).
> (2) [Walkthroughs → `@ttsc/banner`](/docs/development/walkthroughs/banner) — line-by-line walkthrough of the simplest shipped plugin. The friendliest first read for Go newcomers.
> (3) [Authoring → Getting started](/docs/development/authoring/getting-started) — the Hello plugin: bootstrap, printer round-trip, no mutation. Write your first plugin from scratch.
> (4) [Reference → Pitfalls](/docs/development/reference/pitfalls) — skim before you build so the common first-hour failures are recognizable.
> (5) [Authoring → End-to-end transform](/docs/development/authoring/end-to-end) — the same shape with a real AST mutation. Then read [`@ttsc/strip`](/docs/development/walkthroughs/strip), then [`@ttsc/paths`](/docs/development/walkthroughs/paths) for production-quality versions of the same shape.
> (6) [Authoring → Testing](/docs/development/authoring/testing) and [Publishing](/docs/development/authoring/publishing) — after your first plugin runs.

The [`@ttsc/lint` deep dive](/docs/development/walkthroughs/lint) is the last stop after you finish the paths walkthrough, when you are ready to ship a multi-file engine.

## How a plugin works, in one paragraph

A `ttsc` plugin is an npm package with two halves: a **JS descriptor** that lives in the consumer's `node_modules` and tells `ttsc` what the plugin does, and a **Go source plugin** that holds the actual logic. Executable `package main` sources run as sidecars; non-`main` transform packages link into the selected native host. The Go source reads the TypeScript-Go AST, queries the Checker, then either reports diagnostics or rewrites emit. It compiles once the first time it is used, and the result is **cached** by ttsc version, tsgo version, platform, plugin source hash, overlay hash, contributor hashes, and the Go build environment. Plugins run through the `ttsc` CLI, `ttsx`, and every `@ttsc/unplugin` bundler adapter. `ttscserver` exposes the public `driver.PluginSource` seam for editor embedders; the shipped VS Code extension is the reference client for project TypeScript-Go plus LSP-capable plugin diagnostics/actions. One contract, one enablement entry in `compilerOptions.plugins[]`; a `package.json#ttsc.plugin` field on a directly-listed dependency adds **auto-discovery** as a package-side fallback. See [Concepts](/docs/development/concepts) for the canonical vocabulary.

## Status

v1 protocol, still moving. Don't publish `ttsc` as a peer dependency yet — treat it as a build-time-only dependency. The driver surface (`packages/ttsc/driver/`) is the closest thing to a stable plugin SDK; symbols documented in [Reference → Driver API](/docs/development/reference/driver-api) survive a normal release cycle.

## Requirements

- Node.js 18+ — install from [nodejs.org/download](https://nodejs.org/en/download) if you don't have it.
- Go 1.22+ for direct `go test` / `go vet` against your plugin — install from [go.dev/dl](https://go.dev/dl/). `ttsc`'s bundled toolchain (shipped in the platform package) is what actually builds plugins on consumer machines; a system Go is only needed for local-dev workflows. The shipped plugins (and the `go.mod` examples in this chapter) target `go 1.26` because that is the bundled SDK; the bundled toolchain satisfies the `go` directive regardless of your system Go version.
- A consumer test fixture with `ttsc` and `@typescript/native-preview` installed.
