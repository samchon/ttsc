# Code Graph (MCP)

`@ttsc/graph` gets its own benchmark because it measures agent behavior, not compiler speed. The chart on this page asks whether a coding agent can answer a code-tour question with fewer tokens and fewer tool calls when a checker-backed graph is available.

<GraphBenchmark />

## What the Benchmark Measures

The benchmark runs the same question against an empty MCP baseline and against one graph server at a time. The useful signal is whether the agent can stay on the graph long enough to answer from symbols, relationships, and ranges instead of widening into broad file reads.

`@ttsc/graph` is a checker-backed TypeScript graph MCP. It returns compact index evidence, not inline source bodies. For code-tour questions, the `tour` branch returns central entrypoints, primary flow, nearby paths, tests, and read-next anchors as one answer surface.

A separate structural harness checks graph shape and cross-file coverage. That validates the graph itself. This page is about the agent-cost question: does the graph help a normal agent make the next decision with less searching?

## Why `@ttsc/graph` Is Different

The request surface is built around visible reasoning and an explicit stop path. The agent fills `question`, `graphNeed`, `draft`, and `review` before the final request, then either inspects the graph or takes the `escape` branch when graph is not the next useful evidence.

That matters because the benchmark is trying to support normal Codex and Claude Code behavior, not a benchmark-only workflow. The graph should help an agent decide what to read next, not trap it in repeated graph calls or force it into a source-extraction posture.

The shape of the payload matches that goal. `inspect_typescript_graph` returns evidence coordinates and ranges, stable handles for follow-up calls, and a `next` action that says whether to answer, inspect more graph, leave graph, or clarify. For onboarding prompts it can return an answer-ready tour, so the agent does not need to reconstruct that tour from broad file reads.

## Why the Comparator Tools Can Drift

`codegraph` and `codebase-memory` are measured as comparators, not as the same contract. The chart reads raw samples from `website/public/benchmark/graph.json`, so negative bars are not hidden and a comparator can look good or bad depending on how the agent actually used it.

The spread is consistent with a few failure modes. If a tool returns source-like payloads instead of compact handles and ranges, the agent pays for that payload in every later turn. If the tool contract does not make the stop condition explicit, the agent can keep asking for adjacent evidence, repeat tool calls, or fall back to shell reads when the graph result was already enough.

The benchmark keeps those raw outcomes visible. The point is not to teach the user prompt how to favor one product. The point is to check whether the MCP server's own schema, descriptions, and results help the agent preserve its normal judgment while spending less context on search.

## Benchmark Your Project

Start from a clean checkout, build the graph package, and measure the structural graph first. For a local agent-cost smoke run, pass your repository directory and the same natural question to the baseline arm and the graph arm.

```bash
git clone https://github.com/samchon/ttsc.git
cd ttsc
corepack enable
pnpm install
pnpm --filter @ttsc/graph build
node experimental/benchmark/graph/bench.mjs --project=/abs/path/to/project --tsconfig=tsconfig.json --runs=5
node experimental/benchmark/graph/agent-ab-codex.mjs --repo=typeorm --repo-dir=/abs/path/to/project --tsconfig=tsconfig.json --question "Trace the main runtime path I should understand before editing." --arm=baseline --runs=5
node experimental/benchmark/graph/agent-ab-codex.mjs --repo=typeorm --repo-dir=/abs/path/to/project --tsconfig=tsconfig.json --question "Trace the main runtime path I should understand before editing." --arm=graph --runs=5
```
