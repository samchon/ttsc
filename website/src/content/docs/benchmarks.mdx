# Benchmarks

How much faster is `ttsc` than `tsc` + `ts-patch` on a real-world TypeScript backend?

We measured [`@samchon/shopping-backend`](https://github.com/samchon/shopping-backend), a NestJS + nestia + typia + Prisma e-commerce backend, comparing its `legacy` branch (TypeScript 5.9 + `ts-patch` + `typescript-transform-paths` + `typia/lib/transform`) against its `next` branch (`ttsc` + `@ttsc/paths` + `typia` native plugin running through `@typescript/native-preview`). The application code is the same on both branches; only the compiler toolchain changes.

## Results

Three timed runs each, after one untimed warm-up so the typia native plugin sits in its on-disk cache for both stacks.

| Command | Legacy (tsc + ts-patch) | Next (ttsc) | Speedup |
| ------- | -----------------------: | -----------: | ------: |
| `pnpm build:main` | 18 660 / 18 294 / 18 413 ms<br/>median **18 413 ms** | 6 700 / 6 601 / 6 674 ms<br/>median **6 674 ms** | **2.76×** |
| `pnpm build:test` | 18 954 / 19 038 / 19 257 ms<br/>median **19 038 ms** | 7 110 / 7 131 / 6 913 ms<br/>median **7 110 ms** | **2.68×** |

`build:main` runs the full backend through the compiler with the typia transform; `build:test` adds the e2e test directory on top. `build:sdk` is excluded because it shells out to `nestia all` rather than the TypeScript compiler proper.

## What the runs do

- `build:main` &nbsp;→ `rimraf lib && {tsc|ttsc}` &nbsp;— full project compile.
- `build:test` &nbsp;→ `rimraf bin && {tsc|ttsc} -p test/tsconfig.json` &nbsp;— compiles the application plus the e2e test suite.

Both stacks emit `.js`, `.js.map`, and ship typia's runtime transforms inline. `legacy` does this through `ts-patch`'s patched `tsc` pulling `typescript-transform-paths` and `typia/lib/transform`; `next` does it through `ttsc` driving `@typescript/native-preview` (TypeScript-Go) with the `@ttsc/paths` and `typia` native plugins.

## Reproduce

```bash
git clone https://github.com/samchon/shopping-backend
cd shopping-backend

# Legacy
git checkout legacy
pnpm install
time pnpm build:main
time pnpm build:test

# Next
git checkout next
pnpm install
time pnpm build:main
time pnpm build:test
```

Run each timing 3+ times and report the median — first-run cost is dominated by the typia native plugin cold build, which is the same on both stacks once cached.

## Caveats

- These numbers come from a single workstation (WSL2). Absolute milliseconds will vary by hardware; the **ratio** is the durable comparison.
- The `next` measurements run on `ttsc` 0.12.1 with two unreleased driver patches landed on the `feat/benchmark` branch (loose source→output mapping fix and whitespace-tolerant call matching). On stock 0.12.1, the same project's `build:main` errors out on barrel files before timing data is reachable; the fixes are required to make the benchmark possible.
- `nestia all` (the SDK generator) is excluded from this comparison because it runs its own pipeline rather than the TypeScript compiler proper. Migrate it on its own clock.
