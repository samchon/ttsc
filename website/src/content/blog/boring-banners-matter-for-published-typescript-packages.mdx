> ## TL;DR
>
> `@ttsc/banner` adds a fixed JSDoc banner to emitted files:
>
> ```js
> /**
>  * License MIT (c) 2026 Acme
>  *
>  * @packageDocumentation
>  */
> ```
>
> It is not a flashy plugin.
>
> It is the kind of boring package-production detail a compiler host should handle cleanly.

---

## 1. Published Packages Need Polish

Library authors care about details:

- license headers;
- generated-file warnings;
- package documentation comments;
- provenance strings;
- consistent declaration output.

None of those are algorithmically exciting.

They still matter.

If you publish TypeScript packages, emitted JavaScript and `.d.ts` files are your product surface.

Small details show up there.

---

## 2. What @ttsc/banner Does

Install:

```bash
npm install -D @ttsc/banner
```

Register:

```jsonc
{
  "compilerOptions": {
    "plugins": [{ "transform": "@ttsc/banner" }],
  },
}
```

Configure:

```ts
import type { ITtscBannerConfig } from "@ttsc/banner";

export default {
  text: "License MIT (c) 2026 Acme",
} satisfies ITtscBannerConfig;
```

Every emitted file receives a JSDoc package banner.

That includes JavaScript and declaration output.

---

## 3. Why It Is A Compiler Plugin

You could add banners with a post-build script.

Many projects do.

But the compiler pass has useful context:

- which files were emitted;
- where declaration output went;
- how to preserve hashbangs and BOMs;
- how to avoid touching source files;
- how to compose with other emit-time transforms.

The plugin can apply a source preamble before parsing where needed and keep output behavior consistent.

That is cleaner than sweeping a dist directory with string concatenation.

---

## 4. Config Discovery

`@ttsc/banner` looks for:

```text
banner.config.ts
banner.config.cts
banner.config.mts
banner.config.js
banner.config.cjs
banner.config.mjs
banner.config.json
```

If you want an explicit path:

```jsonc
{
  "compilerOptions": {
    "plugins": [
      {
        "transform": "@ttsc/banner",
        "configFile": "./config/banner.config.ts",
      },
    ],
  },
}
```

If no config is found, the compile fails.

That is the right behavior.

A missing banner config should not silently publish unmarked output.

---

## 5. The Common Uses

License:

```ts
export default {
  text: "License MIT (c) 2026 Acme",
};
```

Generated warning:

```ts
export default {
  text: "Generated by Acme SDK. Do not edit by hand.",
};
```

Build provenance:

```ts
export default {
  text: `Built from commit ${process.env.GITHUB_SHA}`,
};
```

The plugin accepts a fixed string.

Interpolation is your build script's job.

That boundary keeps the plugin boring.

---

## 6. Why I Keep Writing About Boring Plugins

Because a plugin host that only supports spectacular demos is not enough.

It also needs to support:

- path alias rewriting;
- debug stripping;
- banners;
- lint;
- format;
- small package-specific transforms.

These are the things that make a toolchain feel complete.

The big story is TypeScript-Go and compiler-powered libraries.

The day-to-day story is often a banner at the top of a `.d.ts` file.

Both matter.

---

## 7. The Takeaway

`@ttsc/banner` is small.

That is good.

It does one package-production job inside the compiler pass and gets out of the way.

That is exactly the kind of plugin a healthy compiler ecosystem should make easy.
