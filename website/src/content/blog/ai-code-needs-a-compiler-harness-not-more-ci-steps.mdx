> ## TL;DR
>
> AI coding makes lint more important, not less.
>
> When code volume goes up, the project needs a stronger harness around generated edits:
>
> ```bash
> npx ttsc --noEmit
> ```
>
> Type errors and lint errors should fail together.
>
> But the harness cannot become a pile of duplicate passes:
>
> ```bash
> tsc --noEmit
> eslint .
> prettier --check .
> custom project checks...
> ```
>
> The whole point of [`@ttsc/lint`](https://ttsc.dev/docs/lint) is to move lint into the TypeScript-Go check path so one compilation pass carries more of the project contract.

---

## 1. Lint Used To Sound Optional

There was a time when lint sounded like taste.

Tabs or spaces.

Semicolons or no semicolons.

Double quotes or single quotes.

People argued about it because the stakes felt low.

That is not the world AI coding creates.

AI agents do not only make style mistakes. They make project-contract mistakes:

- they call deprecated helpers;
- they ignore local conventions;
- they use `any` to escape a type problem;
- they introduce accidental async holes;
- they leave dead branches after a refactor;
- they write code that type-checks but violates the repository's rules.

The human reviewer cannot manually inspect every generated line with the same depth.

So the harness has to get stronger.

---

## 2. The Harness Is The Product Boundary

In an AI-heavy workflow, "the code was generated" is not the interesting event.

The interesting event is:

> Did the generated code survive the repository harness?

That harness includes tests, of course.

But before tests, there is the compiler boundary:

```bash
npx ttsc --noEmit
```

This should answer one question:

> Is this source tree acceptable enough to continue?

TypeScript diagnostics are part of that answer.

Lint diagnostics are part of that answer.

Project-specific plugin diagnostics should be part of that answer too.

That is why I do not want lint to feel like a separate optional ceremony.

I want it to look like a compile error.

---

## 3. Why Separate Passes Hurt More Now

The old stack was already expensive:

```bash
npx tsc --noEmit
npx eslint .
npx prettier --check .
```

In the AI era, that cost compounds.

Agents run checks repeatedly.

Editors run diagnostics repeatedly.

CI runs gates on every branch.

Review bots run their own passes.

If every tool opens the project, parses files, builds partial views of the AST, and reports on the same source tree independently, the harness becomes slower exactly when we need to run it more often.

The answer is not "skip lint."

The answer is "stop paying for the same project shape over and over."

---

## 4. One Compilation Pass Is The Right Unit

TypeScript already has to read the project.

It already has to parse source files.

It already has to build a Program.

It already has to collect diagnostics.

So lint should ask:

> What can I check while that world is already loaded?

That is the `@ttsc/lint` model.

The check path looks like this:

```text
ttsc --noEmit
  -> TypeScript-Go Program
  -> TypeScript diagnostics
  -> native lint diagnostics
```

The format path is separate:

```text
ttsc format
  -> TypeScript-Go AST
  -> format/* rewrites
```

Same package.

Separate commands.

One project contract.

---

## 5. Lint As Compile Errors

The output shape matters.

This:

```text
src/index.ts:1:1 - error TS11966: [no-var] Unexpected var, use let or const instead.
```

is not just a different string format.

It means the lint violation appears where TypeScript errors already appear:

- same terminal gate;
- same CI failure;
- same editor underline;
- same quick-fix lane;
- same project language.

That matters for AI-generated code because every separate report format becomes another integration point.

The harness should be boring.

The code can be generated by a human, an agent, or a script.

The gate should not care.

---

## 6. The Cost Model Has To Be Honest

I do not claim a full `ttsc --noEmit` invocation costs zero.

It still has real work:

- start the native host;
- read `tsconfig.json`;
- load `lint.config.ts`;
- resolve plugins;
- check the plugin binary cache;
- build the TypeScript-Go Program;
- render diagnostics.

But once the Program exists, the lint pass should not build another TypeScript pipeline.

In the VS Code benchmark fixture, the measured `@ttsc/lint time` line can be around hundreds of milliseconds while ESLint takes tens of seconds.

That is the interesting comparison.

Not because milliseconds are magic.

Because the architecture changes where lint lives.

---

## 7. AI Makes Local Rules More Valuable

General-purpose AI models know general TypeScript.

They do not automatically know your repository's contracts.

That is where lint and check plugins become more valuable:

- "never call this helper directly";
- "this framework decorator requires this shape";
- "this package cannot import that package";
- "this generated SDK file must not be edited";
- "this runtime route must have a validator";
- "this public type must satisfy a serialization contract."

Some of those are lint rules.

Some are compiler plugins.

Some are project-specific diagnostics.

They all belong in the same harness.

---

## 8. What I Want The Loop To Feel Like

The loop should be:

```bash
agent edits files
npx ttsc format
npx ttsc --noEmit
run focused tests
```

Format writes.

Check gates.

Tests prove behavior.

No duplicate TypeScript parser if we can avoid it.

No linter that needs its own parallel universe.

No "the editor showed one thing, CI showed another thing."

That is the goal.

---

## 9. The Takeaway

AI coding does not remove the need for static analysis.

It raises the floor.

When code is produced faster, the harness has to reject bad code faster too.

That is why I care about lint as compile diagnostics.

That is why I care about one compilation pass.

And that is why `@ttsc/lint` is not just a faster ESLint clone.

It is a different place to put the gate.
