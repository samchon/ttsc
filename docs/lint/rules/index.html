Rules

A practical rule catalog by category. Each rule maps to an ESLint name where one exists. Severity in lint.config.ts is one of "error" (fails the build), "warning" (prints, exit code 0), "off" (skipped).

If you’re skimming: the error-severity defaults at the bottom of this page are what most projects end up with.

Authoritative source: the canonical rule-name union lives at packages/lint/src/structures/TtscLintRule.ts. This page documents what each rule does and when to keep it on.

Variable declarations

RuleEffect
no-varReject var. Use let or const. Autofixable.
prefer-constWhen a let is never reassigned, demand const. Autofixable for single declarations.

cypress/no-async-beforeReject async before and beforeEach callbacks.
cypress/no-async-testsReject async Cypress test callbacks.
cypress/no-chained-getReject chained .get() calls; use .find() from the previous subject.
cypress/no-debugReject cy.debug() and chained .debug() commands.

cypress/no-pauseReject cy.pause() and chained .pause() commands.
cypress/no-unnecessary-waitingReject numeric cy.wait(...) sleeps.

Architecture boundaries

These rules are inspired by eslint-plugin-boundaries, but @ttsc/lint implements a conservative TypeScript source-path subset. They inspect static import and export ... from module specifiers, resolve relative imports to real source files, and classify files with the rule option’s elements array.

RuleEffect
boundaries/element-typesEnforce allowed dependency directions between configured source-path element types.
boundaries/externalRestrict external package imports by package/specifier pattern.
boundaries/entry-pointRequire imports into an element to target that element’s configured public entry files.
boundaries/no-privateReject imports of configured private files from outside their element.
boundaries/no-unknownReject relative imports whose resolved source file matches no configured element.
// lint.config.ts
export default {
  rules: {
    "boundaries/element-types": ["error", {
      elements: [
        { type: "app", pattern: "src/app/**" },
        { type: "domain", pattern: "src/domain/**", entry: "index.ts", private: "internal/**" },
      ],
      rules: [{ from: "app", disallow: "domain" }],
    }],
    "boundaries/entry-point": ["error", {
      elements: [
        { type: "domain", pattern: "src/domain/**", entry: "index.ts" },
      ],
    }],
    "boundaries/no-private": ["error", {
      elements: [
        { type: "domain", pattern: "src/domain/**", private: "internal/**" },
      ],
    }],
    "boundaries/no-unknown": ["error", {
      elements: [
        { type: "app", pattern: "src/app/**" },
        { type: "domain", pattern: "src/domain/**" },
      ],
    }],
    "boundaries/external": ["error", { disallow: ["@legacy/sdk"] }],
  },
};

These diagnostics are not autofixable. A boundary violation usually needs a public API, a dependency inversion, or a config decision; rewriting the import path automatically would be unsafe.

Functional Policy

The functional/* family is an opt-in strictness pack for projects that want immutable, expression-oriented TypeScript. These rules are intentionally not core defaults because they encode a programming style, not just bug prevention. They are diagnostic-only; ttsc fix does not rewrite architecture or control flow choices automatically.

RuleEffect
functional/functional-parametersReject rest parameters, arguments, and optionally zero-parameter functions.
functional/immutable-dataReject writes through object/array members and mutable collection methods.
functional/no-class-inheritanceReject class inheritance and abstract classes.
functional/no-classesReject class declarations and expressions.
functional/no-conditional-statementsReject if and switch statements.
functional/no-expression-statementsReject expression statements used for side effects.
functional/no-letReject let declarations.
functional/no-loop-statementsReject imperative loop statements.
functional/no-mixed-typesReject type/interface declarations that mix member shapes.
functional/no-promise-rejectReject Promise.reject(...).
functional/no-return-voidReject void returns and void-returning declarations.
functional/no-this-expressionsReject this expressions.
functional/no-throw-statementsReject throw statements.
functional/no-try-statementsReject try/catch/finally statements.
functional/prefer-immutable-typesPrefer readonly/immutable type annotations.
functional/prefer-property-signaturesPrefer function-property signatures over method signatures.
functional/prefer-readonly-typeRequire readonly array, tuple, and property type syntax.
functional/prefer-tacitReport simple one-argument forwarding wrappers.
functional/readonly-typeEnforce the configured readonly type spelling.
functional/type-declaration-immutabilityRequire matching type declarations to expose readonly member shapes.
// lint.config.ts
export default {
  rules: {
    "functional/no-let": "error",
    "functional/no-loop-statements": "error",
    "functional/no-conditional-statements": "error",
    "functional/no-throw-statements": "error",
    "functional/no-try-statements": "error",

Format

The format* rules cover Prettier’s territory. They live on their own page — Format rules.

A common starter set

Most projects start here. Add and remove as you go.

// lint.config.ts
import type { ITtscLintConfig } from "@ttsc/lint";
 
export default {
  rules: {
    // Variables
    "no-var": "error",
    "prefer-const": "error",
 
    // Comparison
    "eqeqeq": "error",
    "no-self-compare": "error",
 
    // Suggestions
    "no-explicit-any": "warning",
    "prefer-as-const": "error",
    "object-shorthand": "error",

Autofix coverage at a glance

ttsc fix autofixes a subset. Native fixers cover:

  • no-var, single-declaration prefer-const, ESLint-safe eqeqeq
  • no-wrapper-object-types (primitives only; Object is detection-only)
  • prefer-as-const, no-useless-rename, object-shorthand, no-extra-non-null-assertion
  • no-unnecessary-type-constraint, prefer-namespace-keyword, no-useless-escape
  • no-import-type-side-effects (hoists inline type modifiers)
  • await-thenable (type-aware — drops await on non-thenable operands)
